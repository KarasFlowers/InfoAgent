1. 复杂环境下的网页精准抓取与质量评估
技术难点：
信息噪音多：技术文章通常夹杂大量的侧边栏、广告、导航栏及脚本代码，直接抓取会导致噪音污染。
反爬与渲染防御：许多站点（如 Medium、Hacker News 引向的外部链接）存在反爬机制。
解决办法：
双引擎提取策略：首选 trafilatura（高精度的正文提取库）进行结构化提取，若其解析失败则降级到 BeautifulSoup 手动清洗。
质量自评体系：实现 assess_content_quality 机制，通过计算文本密度（有效字数占比）、噪声率（广告词命中率）和文本长度得出质量分。
异常拦截：对于质量评分过低的内容，自动触发“内容解析不全”的提示，防止脏数据进入 AI 摘要链。
2. 高精度的 RAG 混合检索管线 (Hybrid Search)
技术难点：
语义与关键词失衡：纯向量检索（Bi-Encoder）有时会丢失专有名词（如特定的库名、版本号），而纯关键词检索（BM25）又不懂语义。
解决办法：
两阶段检索方案：
Recall（召回）：同时使用 Bi-Encoder（语义向量） 和 BM25（传统关键词） 进行召回。
Fusion（融合）：利用 RRF（倒数排名融合） 算法，将两路结果加权合并，确保搜索结果既“懂意思”又“抓得准词”。
Rerank（精排）：引入 Cross-Encoder 系统。在初步筛选出的 Top-20 片段中，使用更大规模的模型进行深度语义对齐评分，最终只选取最相关的 Top-3 喂给 LLM，极大地减少了模型“幻觉”。
3. 基于“用户画像”的个性化重排序
技术难点：
信息过载：即使内容再好，如果不是用户感兴趣的方向（如：一个前端工程师可能对 C++ 内核文章兴趣有限），价值也会大打折扣。
解决办法：
语义反馈学习 (Learning Service)：当用户点击 👍（喜欢）或 👎（不喜欢）时，系统会自动提取该文章的语义向量。
正负质心画像 (Positive/Negative Centroids)：实时维护用户兴趣的“语义质心”。
多因子评分模型：在 RAG 排序阶段，最终分 = 搜索相关性 * 0.7 + 用户兴趣匹配度 * 0.3。通过叠加“兴趣加分”和“噪音扣分”，实现越用越懂你的搜索体验。
4. 系统响应性能优化与异步架构
技术难点：
链路过长：抓取 -> 分块 -> 向量化 -> 检索 -> 排序 -> LLM 生成 全流程耗时极长，用户等待体验差。
解决办法：
流式响应 (Streaming)：不仅在最后的对话环节使用流式输出，还在侧边栏“快速概要”生成时也采用流式展示，让用户在 1s 内就能看到内容。
多级缓存设计：
内存缓存 (LRU)：缓存已处理的文章正文和摘要。
持久化索引：使用 ChromaDB 持久化向量库，相同 URL 的二次访问无需重新计算 Embeddings。
I/O 与 CPU 并行：在 Python 端使用 asyncio 处理 API 与网络请求，使用 asyncio.to_thread 将 CPU 密集型的 Embedding 计算扔进线程池，防止 UI 阻塞。
5. RAG 端数据的冷热分离与持久化
技术难点：
向量库膨胀：如果每个用户看过的文章都全文存入向量库，存储压力巨大且检索性能会随之下降。
解决办法：
按需索引与生命周期管理：系统采用“冷热分离”策略。最新抓取的文章作为“热点”优先保持索引，并通过定时任务清理旧的临时 RAG 集合。
关系型与非关系型混合驱动：使用 SQLite (SQLModel) 存储文章元数据和用户反馈，ChromaDB 负责非结构化的片段检索，互补长短。
