Argos 后续方向（按阶段推进）

当前阶段的核心原则：优先做能直接增强用户体验、且能复用现有架构的产品能力；高投入的新架构方向保留为中远期探索，不作为下一阶段默认主线。

一、近期优先

1. 历史 / 跨天洞察
痛点：现在更像“当天资讯阅读器”，缺少连续观察能力。
做法：增强历史归档、近 7 天 / 30 天回顾、主题变化追踪、同一公司或模型的连续演进观察。
目标：让 Argos 从“看今天”升级为“理解一段时间内发生了什么”。

2. 更强的阅读助手交互
痛点：当前 RAG 已能追问单篇文章，但仍偏“问答框”，还不够像阅读助手。
做法：补充追问建议、引用依据、高亮关键证据、多文章对比入口，以及“为什么这篇值得看”的解释。
目标：让用户在阅读单篇文章时更快获得结构化理解。

3. 显式偏好控制
痛点：现在已有反馈和 persona，但用户对个性化控制感还不够强。
做法：支持关注主题、屏蔽主题、来源偏好、摘要风格偏好，并与现有反馈学习结合。
目标：让推荐结果既会学习，也可被用户主动调节。

4. 异步增量索引管道
痛点：当前文章索引仍偏点击后触发，体验上会感觉“等系统现读”。
做法：在 RSS 抓取到新文章后，后台自动完成抓取、提取、打标、入向量库，为阅读助手提供更快响应。
目标：把它作为支撑体验的基础设施，而不是第一优先的单独产品卖点。

二、中期增强

1. 热点专题簇 / 自动聚类
痛点：当前分类和标签仍偏离散，难以自然形成“今天大家都在谈什么”的专题视角。
做法：基于 embedding 与聚类方法自动发现热点专题，把相关文章归成可浏览的主题簇。
目标：从“文章列表”提升到“主题视图”。

2. 多文章联合理解 / 对比阅读
痛点：用户很难直接比较多篇文章对同一主题的共识、差异与演进。
做法：支持多篇文章联合检索、差异总结、观点对照、时间顺序整理。
目标：让 Argos 逐步具备“研究助手”而不只是“阅读助手”的能力。

三、远期探索

1. GraphRAG
价值：适合处理跨文章实体关系、长期知识结构与复杂推理。
说明：方向正确，但建设成本高，需要知识抽取、关系更新、图检索与图推理协同，暂不作为近期主线。

2. 完整 Agentic RAG
价值：适合让系统主动选择工具、组合检索路径、补充背景信息。
说明：收益高，但复杂度和不确定性也高。近期最多只做“受限工具化”的局部能力，不直接进入全自主 Agent 模式。

3. 多模态理解
价值：对流程图、架构图、截图密集的技术文章很有帮助。
说明：这是清晰的长期增强方向，但前提是先把文本阅读助手体验做扎实。

阶段判断

当前不宜把 GraphRAG、完整 Agentic RAG、多模态理解作为下一步主线。原因不是这些方向不重要，而是它们更适合在现有产品体验已经足够强之后，再作为“能力跃迁”来做。

四、补充方向（2026-04 评审补入）

A. 阅读助手核心增强
  1. Citation 溯源：RAG 回答里插入 [1][2] 标记，点击可在侧栏定位原文 chunk；是“阅读助手”真正的门面。
  2. HyDE / 查询改写：复杂追问先让 LLM 生成假想答案再检索，显著提升中文长问答命中率。
  3. Embedding 升级到 BGE-M3：中文 NDCG 预期提升 10~15%；换模型 + 重建索引一次性工作。

B. 产品外延
  4. ✅ 邮件 / RSS 日报出口：每天定时把简报 push 到邮箱，或把简报本身做成 RSS feed，让它从“主动打开”变“被动接收”。
  5. 简单多用户：登录 + persona/feedback 隔离，支持 2~5 人共用同一实例。
  6. ✅ 成本 / 消耗可视化：今日 token、LLM 延迟 p50/p99，是后续接多 LLM 路由的基础。

C. 工程底座（正交升级）
  7. 前端现代化：把 1300 行 vanilla JS + Jinja 迁到 Vite + 轻型框架（Svelte 或 React）+ TypeScript，为后续 UI 功能解压。
  8. ✅ 结构化日志 + traceid：用 structlog，每请求一个 id 串联 RSS/LLM/DB/RAG 日志，事故定位从分钟降到秒级。
  9. Service 层单测：_score_articles / _recall_and_rerank / rerank_summary_items 等核心算子补测试，改动有保障。
  10. ✅ APScheduler 后台清理：把自动清理从“请求触发”挪到定时任务，避免清理失败污染请求 session 的事故。
  11. Prometheus /metrics：RSS 抓取成功率、LLM 延迟、RAG 命中率、反馈样本数——后续做任何 A/B 的前提。
  12. 向量库可选切换到 Qdrant：Chroma 多进程不稳，多用户引入后会成为瓶颈。

下一阶段推荐动作（更新版）

先做两件"产品感"最强且彼此耦合低的事，并保留"工程底座"作为穿插项：
1. ✅ 异步增量索引管道（通用前置，消除"点击后等待"感）
2. ✅ Citation 溯源 + HyDE 查询改写（阅读助手直接升级）
3. ✅ 历史 / 跨天洞察最小切片（实体时间线 + 话题温度曲线）
4. ✅ 显式偏好控制（扩 UserPersona.category 枚举即可，不建新表）
5. ✅ 工程底座中先做：结构化日志 + APScheduler（为上面几项排雷）
6. ✅ 附加提升：Embedding 升级到 BGE-M3 + 系统消耗可视化（今日 Token / P50 P99 延迟面板）
